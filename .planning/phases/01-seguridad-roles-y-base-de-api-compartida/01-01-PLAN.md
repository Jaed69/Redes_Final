<objective>
Fix the two hardcoded/duplicated JWT secrets (SEC-01) and the missing auth on `POST /empresas` (SEC-02), implement the `requireRole(...roles)` authorization middleware (SEC-04), apply it to every existing mutation across all 11 resource route files, and close a discovered password-hash-leak in the Usuario update endpoint.

Purpose: this is the backend half of the phase's security fix — it establishes the single mechanism (`requireRole`) that every future phase's Admin/Monitor/Cliente/TI mutation gating depends on, and removes the two concrete, currently-exploitable holes (forgeable/mismatched JWTs, unauthenticated empresa creation).
Output: `auth.middleware.js` exporting `{ verificarToken, requireRole }`, `auth.service.js` reading the same `JWT_SECRET`, all 11 route files gating their mutations by role, `usuario.service.js` no longer leaking password hashes.
</objective>

    In `auth.middleware.js`: replace the module-level `SECRET_KEY` assignment (currently a hardcoded quoted string) with `const SECRET_KEY = process.env.JWT_SECRET;` — same line position, same const name, no other line changes to `verificarToken`. Add a new `requireRole` export in the same file, following `verificarToken`'s exact style (a plain closure, same `{ mensaje }` JSON error envelope used everywhere else in this codebase per CLAUDE.md conventions): `requireRole` takes a variadic list of allowed role strings and returns an Express middleware `(req, res, next)`. It must fail closed — if `req.usuario` is missing (defensive: verificarToken should always have set it first, but do not assume) OR `req.usuario.rol` is not included in the allowed list, respond `403` with `{ mensaje: "No autorizado para esta acción" }` and do not call `next()`; otherwise call `next()`. Change the file's final `module.exports` from the bare `verificarToken` function to an object `{ verificarToken, requireRole }` — this is a breaking export-shape change that Task 2 (this same plan) and every route file depend on.

    In `auth.service.js`: replace its own separately-hardcoded `SECRET_KEY` (currently a second, independently-duplicated literal near the `login` function) with `const SECRET_KEY = process.env.JWT_SECRET;`. Do not touch the `jwt.sign(...)` call itself or the `rol` claim — those already flow the DB's `rol` value straight through the token and response body, which is exactly what the 4 short role codes (`admin`/`cliente`/`monitor`/`ti`, per CONTEXT.md D-01) need; no service-layer logic changes are required beyond the secret source.

    Do not introduce a shared constants file or import between the two files — per PATTERNS.md, both processes independently read the same `process.env.JWT_SECRET`, matching the codebase's existing per-file env-var convention (no shared config module for this).
