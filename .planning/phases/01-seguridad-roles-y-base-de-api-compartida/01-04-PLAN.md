<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser URL bar / direct navigation → React Router route tree | Untrusted — any visitor, authenticated or not, can type any path directly, bypassing whatever UI links would normally lead them there |
| `RequireRole` (client) vs. `requireRole` (server, Plan 01-01) | Two different mechanisms with the same name-shape but different trust levels — the client one is UX, the server one is the actual security boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-07 | Elevation of Privilege | `RequireRole.tsx` (client-side route guard) | medium | accept | Client-side guards are enforceable only within the browser and can always be bypassed by editing `localStorage` directly or calling the API with curl/devtools, bypassing the UI entirely — this is inherent to any SPA route guard, not a defect in this implementation. The authoritative security boundary is the backend `requireRole` middleware (Plan 01-01), which independently rejects any mutation from a non-admin token regardless of what the frontend renders or believes. `RequireRole` exists to keep legitimate users out of the wrong role's UI, not to stop a determined attacker; documented here as an accepted, inherent limitation rather than a gap to close. |
| T-01-SC | Tampering (supply chain) | npm installs | n/a | accept | No new package installs — `react-router-dom` is already a dependency, no axios/UI-library addition |
</threat_model>
