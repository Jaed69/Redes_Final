  <action>
    Implement `services/api.ts` as a module with a single named export `export const api = { get, post, put, del }` — this exact shape is a hard contract: Plan 01-04 imports it as `import { api } from "../services/api"` (or the equivalent relative path) and calls `api.get`/`api.post`/`api.put`/`api.del` directly, so do not switch to a default export or to loose top-level named exports (`export function get`, etc.) instead of the single `api` object. Each of the four methods is generic over a response type and returns `Promise<T>`.

    Base URL: `const BASE_URL = import.meta.env.VITE_API_URL || "http://localhost:3000";` — read once at module scope, matching the fallback value already hardcoded across the codebase today so nothing breaks for developers who have no `.env` set for the web app.

    Internally, all four methods should funnel through one private `request<T>(path, options)` helper that: builds the full URL as `` `${BASE_URL}${path}` ``; always reads the Bearer token via `localStorage.getItem("token")` and attaches `Authorization: Bearer <token>` when a token is present (omit the header entirely when there is none, rather than sending `Bearer null`); sets `Content-Type: application/json` only when a body is being sent (post/put); JSON-stringifies the body for post/put; calls `fetch`; parses the JSON response body; and — this is the key behavior change other plans depend on — if `response.ok` is false, throws `new Error(data.mensaje || "Error de red")` instead of returning the raw `Response`, so callers can keep using their existing `try { ... } catch (error) { alert(error.message) }` pattern without ever touching `response.ok` themselves. On success, return the parsed JSON body directly (matching the current `.then(r => r.json())` shape every existing caller already expects).

    Do not add any new npm dependency (no axios) — plain `fetch` is already globally available in the Vite/browser environment this code runs in.
  </action>
